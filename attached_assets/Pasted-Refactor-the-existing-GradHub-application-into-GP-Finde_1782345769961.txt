3. PRIVACY STRATEGY (Extend existing PublicViewStrategy / ConnectedViewStrategy)
   - Phone and GPA are NEVER included in any API response for another user unless:
     * The requesting user is viewing their OWN profile (always see full data), OR
     * An accepted ConnectRequest currently exists between the requesting user and the target user.
   - Enforce this server-side in the PrivacyMaskService / Strategy, not just UI-hidden. Return null/omit the fields entirely.
   - For project applications while still pending: the leader sees applicant's name, track, skills, bio, bylaw, and gender — never GPA or phone, in any form (no exact value, no bracket/range). This only changes once the application is accepted, per section 4.

4. PROJECT LEADERSHIP
   - Project creator is automatically assigned as leader (leader_id).
   - Only the leader sees Accept/Reject/Remove controls on ProjectApplications for their project.
   - Leader can change member status: 'pending' → 'accepted' or 'rejected'; 'accepted' → 'removed'.
   - Members can self-remove: 'accepted' → 'left'.
   - Application-to-connection bridge: when a leader changes a ProjectApplication's status from 'pending' to 'accepted', the system automatically grants a connection between leader_id and applicant_id:
     * Set requester_id = applicant_id, recipient_id = leader_id, status = 'accepted'.
     * If ANY ConnectRequest already exists between these two users in any status (pending, rejected, etc.), update that single existing row to 'accepted' instead of creating a new one. (This intentionally overwrites any prior personal connection state between the two — project-based contact sharing is the primary connection mechanism on this platform.)
     * This reveals phone and GPA between the leader and that specific applicant only. The applicant's act of applying is the consent for this; rejecting an application never triggers it, and it does not extend to other members of the same project — they remain unconnected to each other and to the leader unless they separately send and accept a ConnectRequest.
   - Revocation on exit: when a ProjectApplication's status changes to 'left' or 'removed', automatically update the ConnectRequest between that member and the leader to 'rejected', revoking the privacy reveal. The member and leader are no longer connected, and phone/GPA become hidden from each other again.
   - Project status transitions: when the first ProjectApplication is accepted, automatically change project.status from 'Open' to 'In Progress'. While 'In Progress', the leader can manually change status to 'Closed' (blocks new applications). Status can never move back to 'Open' once it has left that state.
